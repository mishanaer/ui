# UXW Principles

## Useful

Write only copy that helps the user act, understand a state, avoid an error, or make a decision. If the copy repeats what the component already shows, suggest removing it.

Do:
- explain consequences before irreversible actions;
- say what changed after the action;
- give recovery steps for blocked states;
- mention waiting time only when it is meaningful and known.

Do not:
- describe obvious UI mechanics;
- announce future steps too early;
- add filler such as "для вашего удобства";
- hide product or system limitations behind vague optimism.

## Clear

Use one term for one concept. Prefer concrete nouns and verbs over abstract process words.

Prefer:
- "Заявка отправлена" over "Осуществлена отправка заявки";
- "Мы проверим документы до 18:00" over "Проверка документов будет выполнена в ближайшее время";
- "Не загрузилось. Попробуйте снова" over "Произошла непредвиденная ошибка".

## Simple

Shorter is better only when the result stays clear. If the user needs a consequence or condition, keep it.

Cut:
- duplicate nouns from heading and subheading;
- "можно", "нужно", "необходимо", "следует" when they add no meaning;
- "в целях", "в связи с", "по причине" in ordinary UI copy.

## Neutral Tone

Neutral does not mean cold. Be direct, calm, and helpful.

Avoid:
- jokes in transactional UI;
- "Ой", "Ура", "Супер", "К сожалению";
- compliments that do not help the task;
- apologies in errors unless a specific support or service context requires it.

Use lowercase `вы` in ordinary client UI unless a product style guide says otherwise.
